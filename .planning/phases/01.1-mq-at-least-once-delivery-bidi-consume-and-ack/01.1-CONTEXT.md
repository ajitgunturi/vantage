# Phase 01.1: MQ At-Least-Once Delivery — Bidi Consume + Ack - Context

**Gathered:** 2026-06-28
**Status:** Ready for planning

<domain>
## Phase Boundary

Upgrade the MQ delivery contract from server-stream **at-most-once** (a message
is gone the moment `stream.Send` hands it to the gRPC flow-control window) to
broker-side **at-least-once**: a message is removed only when the consumer
**acks** it; unacked in-flight messages are **re-queued and redelivered** when
the consumer's stream closes.

**Trigger (reproduced defect):** producing 1000 messages then consuming with a
short-lived client (`mqprobe -n 20`) silently lost ~493 messages —
`consumed_total` jumped to 513 while the client read 20 — because the eager
dispatch drains the ring into `workCh` (cap 1024) and `stream.Send` buffers into
gRPC's flow-control window; those in-flight messages die on consumer disconnect.

**In scope:** bidi `Consume` (server streams messages, client streams
credit + acks); broker-assigned per-message id; per-consumer in-flight (lease)
table; credit-based flow control (no eager over-pull); re-enqueue-on-disconnect
redelivery; reworked inspect counters; updated `mqprobe`, README, and the
Phase-1 race tests that this supersedes.

**Out of scope:** disk persistence / WAL (still Phase 6); lease/visibility
timeout for connected-but-stalled consumers (deferred); dynamic credit windows;
consumer groups / replay.

</domain>

<decisions>
## Implementation Decisions

### Delivery semantics
- **D-01:** Delivery becomes **at-least-once**. A message leaves the broker only
  on **ack**; duplicates are allowed (that's the definition) and are absorbed
  downstream by Phase 2 `DB-04` unique constraint + Phase 3 `COLL-05`
  `ON CONFLICT` upsert (the idempotency layer built upfront for exactly this).
- **D-02:** **Unique-delivery-in-steady-state is preserved** (MQ-03): a message
  leased to consumer A is not delivered to B unless A disconnects without
  acking, at which point it is redelivered (possibly to B). No duplication
  absent a disconnect.

### Ack transport — bidirectional stream
- **D-03:** `Consume` becomes a **bidirectional** stream. Server → client streams
  delivered messages; client → server streams **credit + acks**. (Chosen over a
  separate unary `Ack` RPC because client-driven credit also fixes the eager
  over-pull at its root.)

### Redelivery trigger — disconnect-only
- **D-04:** Re-enqueue a consumer's **unacked leases** to the store when its
  stream closes (`ctx.Done`). **No lease/visibility timeout** in this phase — no
  sweeper goroutine, no injected clock. (Timeout for connected-but-stalled
  consumers is a deferred enhancement; see Deferred Ideas.)
- **D-05:** Re-enqueued messages go to the **front** of the ring (oldest-first
  redelivery) so redelivery preserves approximate order; at-least-once permits
  out-of-order, so this is best-effort, not a guarantee.

### Message identity — id on the payload
- **D-06:** Add a broker-assigned **monotonic `uint64 id`** field to
  `TelemetryMessage` (new field number; e.g. `id = 7`). The broker owns it;
  Streamer restamp ignores it. Ack references this id. (Chosen over a `Delivery`
  envelope to minimize consumer call-site churn.)

### Flow control — initial credit + replenish-per-ack
- **D-07:** Client opens the stream with an **initial credit `C`**; the server
  keeps **outstanding-unacked ≤ C** per consumer and sends the next message only
  as acks free slots (one credit returned per ack). Bounded in-flight window;
  no eager 1024-deep `workCh` sweep. (Chosen over explicit dynamic `AddCredit(n)`
  for minimal protocol state.)
- **D-08:** `C` is configurable (env, like `MQ_BUFFER_SIZE`/`WorkChCap`) with a
  sane default; planner/researcher pick the default. The eager single-dispatch →
  shared-`workCh` model is **replaced** by demand-driven, credit-gated delivery.

### Inspect counters (control plane)
- **D-09:** `consumed_total` now counts **acks** (confirmed deliveries), not
  Sends. Add **`delivered_total`** (messages Sent, may exceed acks) and
  **`redelivered_total`** (re-enqueued on disconnect). Optionally an `in_flight`
  gauge. `GET /api/v1/queue/inspect` JSON shape changes — update its consumers
  (smoke script `sed` parsers, README sample, http tests).

### TDD (tests are the spec)
- **D-10:** Race tests (`go test -race`) must, at minimum: (a) **reproduce the
  1000-produce / short-consumer loss against the OLD behavior intent and prove
  it no longer loses** under the new engine; (b) prove **no over-pull** — a
  consumer with credit `C` never has > `C` unacked in flight; (c) prove
  **redelivery-on-disconnect** — unacked leases of a dropped consumer are
  redelivered to a survivor with zero loss; (d) prove **unique delivery across K
  consumers in steady state** is preserved (MQ-03 still holds when everyone
  acks). Keep the existing `-count=50` race discipline.

### Phase 6 reconciliation — deferred
- **D-11:** 01.1 delivers **live-path** at-least-once; Phase 6 WAL remains framed
  as-is **for now**. The overlap (Phase 6 SC "at-least-once via replay") is noted
  here and in STATE; Phase 6 narrows to **crash-durability only** when it is
  planned (`/gsd-plan-phase 6`). No pre-emptive ROADMAP edit to Phase 6 in this
  phase.

### Claude's Discretion
- Exact new proto field numbers, the `AckOrCredit` client→server message shape,
  default credit `C`, internal lease-table data structure, and counter field
  naming — planner/researcher decide within the decisions above.

</decisions>

<specifics>
## Specific Ideas

- The smoke probe `-mode consume -n K` must stop being a misleading "peek": under
  bidi+credit it should pull at most `C` unacked and ack what it reads, leaving
  the remainder buffered (not silently discarded).
- Keep the late-join smoke scenario — it stays valid and should now also assert
  no loss when the consumer reads fewer than produced.

</specifics>

<deferred>
## Deferred Ideas

- **Lease / visibility timeout** (redeliver if a *connected* consumer doesn't ack
  within N seconds) — needs a sweeper goroutine + injectable clock. Deferred; can
  be a follow-up enhancement once disconnect-based redelivery lands.
- **Dynamic credit windows** (`AddCredit(n)`) — deferred in favor of
  initial-credit + replenish-per-ack.
- **Phase 6 WAL re-scope to crash-durability-only** — do at `/gsd-plan-phase 6`,
  not now (D-11).
- **Graceful drain of in-flight on shutdown** (ENH-04) — related but separate.

None of these block this phase.
