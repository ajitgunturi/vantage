## User Constraints (from CONTEXT.md)

### Locked Decisions

**Delivery semantics**
- D-01: Delivery becomes at-least-once. A message leaves the broker only on ack; duplicates absorbed downstream by Phase 2 DB-04 + Phase 3 COLL-05 idempotency layer.
- D-02: Unique-delivery-in-steady-state preserved (MQ-03): a leased message is not delivered to a second consumer unless the first disconnects without acking.

**Ack transport**
- D-03: `Consume` becomes a bidirectional stream. Server→client streams delivered messages; client→server streams credit + acks.

**Redelivery trigger**
- D-04: Re-enqueue unacked leases on `ctx.Done` only. No lease/visibility timeout, no sweeper goroutine.
- D-05: Re-enqueued messages go to the **front** of the ring (best-effort ordering; at-least-once permits out-of-order).

**Message identity**
- D-06: Add broker-assigned monotonic `uint64 id` field to `TelemetryMessage`. Broker owns it; Streamer ignores it.

**Flow control**
- D-07: Client opens with initial credit `C`. Server keeps outstanding-unacked ≤ C per consumer; replenish-per-ack.
- D-08: `C` is configurable (env var, like `MQ_BUFFER_SIZE`/`WorkChCap`), with a sane default.

**Inspect counters**
- D-09: `consumed_total` = acks. Add `delivered_total` (Sends) and `redelivered_total` (re-enqueued on disconnect).

**TDD**
- D-10: Race tests (`-race -count=50`) must cover: (a) no-loss vs old behavior; (b) no over-pull (> C unacked); (c) redelivery-on-disconnect; (d) unique delivery across K consumers in steady state.

**Phase 6 reconciliation**
- D-11: Phase 6 WAL narrows to crash-durability-only when planned. No pre-emptive changes now.

- This phase overrides `instructions.md` server-side-streaming clause (ADR-001, already created and accepted). The planner's Wave 0 task confirms ADR-001 exists and REQUIREMENTS.md is already updated (both are done — verified in CONTEXT.md and repo).

## Summary

Phase 01.1 upgrades the MQ delivery contract from at-most-once (messages lost when a consumer disconnects mid-window) to broker-side at-least-once (a message is removed from the broker's custody only when the consumer acks it by broker-assigned id). The root defect is architectural: the single dispatch goroutine eagerly drains the ring buffer into a shared `workCh` (cap 1024), and `stream.Send` on a server-stream hands messages into gRPC's flow-control window where they silently disappear on consumer disconnect. The fix requires eliminating `workCh` entirely and replacing the eager dispatch model with demand-driven, credit-gated delivery directly in each Consume handler.

The four load-bearing changes are: (1) upgrade `api/proto/mq.proto` — add `uint64 id = 13` to `TelemetryMessage`, define a new `ConsumeClientMsg` message, and change `Consume` to a bidi RPC; (2) rewrite `internal/server/server.go` — remove the dispatch goroutine and `workCh`, add a per-consumer recv goroutine inside each `Consume` handler, implement a credit semaphore, and maintain a per-consumer lease table that is re-enqueued on disconnect; (3) extend `internal/queue/store.go` and `ring_store.go` with a `Requeue([]*pb.TelemetryMessage)` method that inserts at the front of the ring; and (4) update `internal/http/inspect.go`, `internal/server/server_test.go`, `scripts/smoke/mqprobe/main.go`, and `scripts/smoke/phase01-mq.sh` for the new bidi contract and counter names.

No new Go dependencies are required. All changes are within the existing Go module. The proto toolchain (raw `protoc` + `protoc-gen-go` + `protoc-gen-go-grpc`) is already installed.

**Primary recommendation:** Eliminate the shared `workCh` + dispatch goroutine pattern entirely. Each Consume handler is fully self-contained: it polls the store under credit control, maintains its own lease table, and re-enqueues on disconnect. A close-and-reopen notification channel in MQServer replaces the old `notify chan struct{}` to broadcast "new messages available" to all waiting consumers.

### Pattern 1: Bidi Stream Handler — Two-Goroutine Structure

**What:** One goroutine exclusively calls `Recv()` (the "recv goroutine"); the handler goroutine exclusively calls `Send()` (the "send loop"). Concurrent Send+Recv on the same stream is safe in gRPC-go; concurrent calls to the same method from multiple goroutines are not.

**When to use:** Always in bidi streaming handlers. This is the only safe pattern per the gRPC-go concurrency contract. [CITED: github.com/grpc/grpc-go/blob/master/Documentation/concurrency.md]

**Example:**
```go
// Source: grpc-go concurrency documentation + this codebase pattern
func (s *MQServer) Consume(stream pb.MQService_ConsumeServer) error {
    atomic.AddInt32(&s.activeC, 1)
    defer atomic.AddInt32(&s.activeC, -1)

    ctx := stream.Context()

    // --- Step 1: read initial credit message ---
    initMsg, err := stream.Recv()
    if err != nil {
        return err
    }
    credit := int(initMsg.GetCredit())
    if credit <= 0 {
        credit = defaultConsumeCredit // 20
    }

    // --- Step 2: credit semaphore (buffered channel acts as token bucket) ---
    sem := make(chan struct{}, credit)
    for i := 0; i < credit; i++ {
        sem <- struct{}{}
    }

    // --- Step 3: per-consumer lease table (owned exclusively by this goroutine's send loop) ---
    leases := make(map[uint64]*pb.TelemetryMessage, credit)

    // --- Step 4: ack channel from recv goroutine to send loop ---
    ackCh := make(chan uint64, credit)

    // --- Step 5: recv goroutine (only goroutine that calls stream.Recv) ---
    var wg sync.WaitGroup
    var recvFinalErr error
    wg.Add(1)
    go func() {
        defer wg.Done()
        defer close(ackCh) // closing ackCh signals send loop to exit
        for {
            msg, err := stream.Recv()
            if err != nil {
                recvFinalErr = err
                return
            }
            if id := msg.GetAckId(); id != 0 {
                select {
                case ackCh <- id:
                case <-ctx.Done():
                    return
                }
            }
        }
    }()

    // --- Step 6: deferred requeue on disconnect (runs after send loop exits) ---
    defer func() {
        wg.Wait() // CRITICAL: join recv goroutine before returning (prevents goroutine leak)
        if len(leases) > 0 {
            msgs := sortedByID(leases)
            s.store.Requeue(msgs)
            atomic.AddInt64(&s.redelivered, int64(len(msgs)))
        }
        // Drain ackCh for any acks that arrived after send loop exited
        for id := range ackCh {
            delete(leases, id)
            atomic.AddInt64(&s.consumed, 1)
        }
    }()

    // --- Step 7: send loop (only goroutine that calls stream.Send) ---
    for {
        // Non-blocking ack drain first
    ackDrain:
        for {
            select {
            case id, ok := <-ackCh:
                if !ok {
                    return recvFinalErr // recv goroutine done
                }
                delete(leases, id)
                sem <- struct{}{} // replenish credit
                atomic.AddInt64(&s.consumed, 1)
            default:
                break ackDrain
            }
        }

        // Acquire credit (blocks if consumer window is full)
        select {
        case <-sem:
            // have credit — fall through to dequeue
        case id, ok := <-ackCh:
            if !ok {
                return recvFinalErr
            }
            delete(leases, id)
            sem <- struct{}{}
            atomic.AddInt64(&s.consumed, 1)
            continue
        case <-ctx.Done():
            return ctx.Err()
        }

        // Poll store for next message
        msg, ok := s.store.TryDequeue()
        if !ok {
            sem <- struct{}{} // return unused credit
            // Wait for new message arrival (or ack, or shutdown)
            notify := s.notifyChan()
            select {
            case <-notify:      // Produce signaled new message
            case <-ackCh:       // ack available; outer loop will drain it
            case <-ctx.Done():
                return ctx.Err()
            }
            continue
        }

        // Assign broker ID and lease the message
        id := atomic.AddUint64(&s.nextID, 1)
        msg.Id = id
        leases[id] = msg

        if err := stream.Send(msg); err != nil {
            // Send failed — re-enqueue this message immediately
            sem <- struct{}{}
            delete(leases, id)
            s.store.Requeue([]*pb.TelemetryMessage{msg})
            return err
        }
        atomic.AddInt64(&s.delivered, 1)
    }
}
```

### Pattern 2: Close-and-Reopen Broadcast Notification Channel

**What:** Replaces the single-consumer `notify chan struct{}` and the `dispatch` goroutine. Broadcasting "store has new messages" to ALL waiting Consume handlers simultaneously using a channel-close as a fan-out primitive.

**When to use:** Any time multiple goroutines must be woken simultaneously by a single event (channel close wakes all readers; the old buffered-1 channel woke only one).

**Example:**
```go
// Source: [ASSUMED] - idiomatic Go broadcast pattern, cross-checked against codebase patterns

type MQServer struct {
    // ... existing fields ...
    notifyMu sync.RWMutex   // guards notifyCh
    notifyCh chan struct{}   // closed on each Produce; immediately replaced
    nextID   uint64          // atomic: monotonic broker-assigned message ID
    delivered  int64         // atomic: messages sent (before ack)
    redelivered int64        // atomic: re-enqueued on disconnect
    // consumed is now acks received (D-09)
}

func NewMQServer(s queue.Store, workChCap int) *MQServer {
    srv := &MQServer{
        store:      s,
        notifyCh:   make(chan struct{}),
        shutdownCh: make(chan struct{}),
        // NO workCh — eliminated entirely
        // NO dispatch goroutine — Consume handlers poll directly
    }
    return srv
}

// notifyAll wakes all goroutines waiting in notifyChan().
// Called by Produce after Enqueue. Lock-free for callers after initial setup.
func (s *MQServer) notifyAll() {
    s.notifyMu.Lock()
    old := s.notifyCh
    s.notifyCh = make(chan struct{})
    s.notifyMu.Unlock()
    close(old) // closing wakes all Consume goroutines blocked on <-notify
}

// notifyChan returns the current notification channel.
// Consume goroutines block on this when the store is empty.
func (s *MQServer) notifyChan() <-chan struct{} {
    s.notifyMu.RLock()
    defer s.notifyMu.RUnlock()
    return s.notifyCh
}

func (s *MQServer) Produce(ctx context.Context, req *pb.ProduceRequest) (*pb.ProduceResponse, error) {
    if req.GetMessage() == nil {
        return nil, status.Error(codes.InvalidArgument, "message must not be nil")
    }
    s.store.Enqueue(req.Message)
    atomic.AddInt64(&s.produced, 1)
    s.notifyAll() // wake ALL consumers, not just one
    return &pb.ProduceResponse{Accepted: true}, nil
}
```

### Pattern 3: RingStore Front-Insertion (Requeue)

**What:** `Requeue([]*pb.TelemetryMessage)` inserts messages at the head of the ring (decrementing the head pointer) to honor D-05 (oldest-first redelivery). If the buffer is full, the tail (newest) entry is dropped to prioritize redelivery. [ASSUMED]

**When to use:** On consumer disconnect — all unacked leases in ID order are re-enqueued at the front.

**Example:**
```go
// Source: [ASSUMED] - derived from RingStore ring buffer invariants in ring_store.go

// Requeue inserts msgs at the HEAD of the ring (front-insertion per D-05).
// Messages should be passed in ascending ID order; they will be delivered in that order.
// If the ring is full, the TAIL (newest) entry is dropped to make room.
// Implements Store.Requeue — added to the Store interface.
func (r *RingStore) Requeue(msgs []*pb.TelemetryMessage) {
    if len(msgs) == 0 {
        return
    }
    r.mu.Lock()
    defer r.mu.Unlock()
    // Insert in reverse order: last element goes in first,
    // so after all insertions the first element is at head (FIFO order preserved).
    for i := len(msgs) - 1; i >= 0; i-- {
        if r.count == r.cap {
            // Drop newest (tail-1) to prioritize redelivery
            r.tail = (r.tail - 1 + r.cap) % r.cap
            r.buf[r.tail] = nil
            r.count--
            r.dropped++ // counts toward dropped even in requeue path
        }
        r.head = (r.head - 1 + r.cap) % r.cap
        r.buf[r.head] = msgs[i]
        r.count++
    }
}
```

### Pattern 5: Proto Changes

**What:** Three changes to `api/proto/mq.proto`. Field numbers are stable; adding field 13 is backward compatible. [CITED: protobuf.dev/programming-guides/proto3]

```protobuf
// Source: api/proto/mq.proto — additions highlighted

message TelemetryMessage {
    string timestamp   = 1;
    string metric_name = 2;
    string gpu_id      = 3;
    string device      = 4;
    string uuid        = 5;
    string model_name  = 6;
    string hostname    = 7;
    string container   = 8;
    string pod         = 9;
    string namespace   = 10;
    double value       = 11;
    string labels_raw  = 12;
    // Broker-assigned monotonic delivery ID (D-06). Set by MQServer on dequeue;
    // zero from Streamer (ignored by broker; overwritten on assign).
    // Field 13 = 1-byte encoding (fields 1-15 are 1-byte; 16+ are 2-byte).
    uint64 id          = 13;
}

// ConsumeClientMsg is streamed client→server on the bidi Consume RPC.
// First message from client: Credit > 0, ConsumerId set, AckId = 0.
// Subsequent messages: AckId > 0, Credit = 0, ConsumerId = "".
message ConsumeClientMsg {
    uint64 ack_id     = 1;  // broker-assigned ID being acknowledged; 0 on initial credit msg
    int32  credit     = 2;  // initial in-flight window; meaningful only on first message
    string consumer_id = 3; // identifies this consumer instance (logging only)
}

service MQService {
    rpc Produce(ProduceRequest) returns (ProduceResponse);

    // Consume is a bidirectional stream (D-03, ADR-001).
    // Server→client: delivers TelemetryMessage with broker-assigned id.
    // Client→server: streams ConsumeClientMsg (initial credit then per-message acks).
    // At-least-once: message is removed from broker custody only on ack.
    // Redelivery on disconnect: unacked leases are re-enqueued at the front of the ring.
    rpc Consume(stream ConsumeClientMsg) returns (stream TelemetryMessage);
}

// ConsumeRequest is retained to avoid breaking generated code referencing it,
// but is no longer used as the Consume RPC input.
// It can be removed in a future cleanup phase.
```

### Anti-Patterns to Avoid

- **Keeping the shared `workCh` + dispatch goroutine:** This is the root of the at-most-once bug. Removing `workCh` is not optional — it must be eliminated. Any design that keeps a shared channel between dispatch and consumers has the same lost-in-transit defect.
- **Calling `stream.Send` from the recv goroutine (or `stream.Recv` from the send loop):** Violates the gRPC-go concurrency contract. Multiple goroutines calling the same method on a stream is a data race. [CITED: github.com/grpc/grpc-go/blob/master/Documentation/concurrency.md]
- **Not joining the recv goroutine before returning from Consume:** The handler function returning causes the gRPC framework to release the stream. Any goroutine still calling `stream.Recv()` after the handler returns will race on a freed stream. Use `wg.Wait()` in the deferred cleanup. [CITED: github.com/grpc/grpc-go/issues/6457]
- **Mutex on the lease table shared between recv goroutine and send loop:** Unnecessary if ack IDs flow from recv goroutine → send loop via `ackCh`. The send loop is the sole writer AND reader of `leases`. Prefer channel communication over shared mutex for this case.
- **Using `sync.Cond` for notify-all:** `sync.Cond.Wait()` does not integrate with `<-ctx.Done()`. Use the close-and-reopen channel pattern instead (Pattern 2).
- **Adding `Requeue` to `RingStore` without adding it to the `Store` interface:** The WAL backend (Phase 6) must also implement `Requeue`. Add it to the interface first.
- **`consumed_total` counting Sends instead of acks after this phase:** The old `consumed++` was at `stream.Send`. After this phase, `consumed` = acks received. Any test that asserts `consumed_total = N after N sends` must be updated (they now assert `consumed_total = N after N acks`).

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Credit semaphore | Custom counter + mutex | Buffered channel as token bucket `make(chan struct{}, C)` | Channels integrate natively with `select` and `ctx.Done`; no mutex deadlock risk; already idiomatic Go |
| Broadcast notification to N goroutines | Mutex-protected slice of channels | Close-and-reopen `notifyCh chan struct{}` (Pattern 2) | Channel close is the native Go fan-out primitive; zero extra goroutines; one RWMutex for the pointer |
| Goroutine synchronization between recv + send | Polling loop with `time.Sleep` | `ackCh chan uint64` + `sync.WaitGroup` | Channels are the right abstraction; WaitGroup ensures clean join |
| Monotonic broker IDs | UUID or timestamp | `atomic.AddUint64(&s.nextID, 1)` | Counter is lock-free; uint64 is the proto type; no collision possible; simpler than UUID |
| Front-insertion in ring buffer | Second queue or linked list | Decrement `head` pointer in RingStore.Requeue | The ring buffer already handles modular arithmetic; inverting the enqueue direction is 10 lines |

**Key insight:** The entire at-least-once upgrade is achievable with Go stdlib only (channels, sync/atomic, sync.RWMutex). Reaching for a third-party library here would add a dependency without solving anything the stdlib cannot handle.

## Common Pitfalls

### Pitfall 1: Goroutine Leak in Recv Goroutine on Handler Return

**What goes wrong:** The Consume handler returns (e.g., `ctx.Done()`), but the spawned recv goroutine is still blocked on `stream.Recv()`. After the handler returns, the gRPC framework tears down the stream, causing the recv goroutine to race on a freed object or leak indefinitely.

**Why it happens:** Missing `wg.Wait()` before the handler returns. [CITED: github.com/grpc/grpc-go/issues/6457]

**How to avoid:** Always `defer wg.Wait()` (or equivalent join) in the Consume handler. The recv goroutine must finish before the handler's stack frame is reclaimed.

**Warning signs:** `TestMQ_GoroutineLeak` fails or shows `runtime.NumGoroutine()` growing with each test iteration under `-race -count=50`.

### Pitfall 2: Lock-Across-Channel Send

**What goes wrong:** The `store.mu` mutex is held while sending to `sem` (credit channel). If the sem is full (buffered), the send blocks while holding the mutex, preventing other goroutines from calling `TryDequeue()` or `Enqueue()`. Deadlock.

**Why it happens:** Forgetting the existing Phase 1 lock discipline: "Channel sends happen with NO mutex held." This discipline is documented in `internal/server/server.go` package comment.

**How to avoid:** Release the store mutex (via `defer` or explicit unlock) before any channel operation. In the new code, `TryDequeue()` returns with the mutex released (it uses `defer r.mu.Unlock()`); never call a channel send inside a function that holds `r.mu`. [CITED: internal/queue/ring_store.go]

**Warning signs:** Deadlock detector fires. `go test -race -count=50` hangs without completing.

### Pitfall 3: AckId Not in Consumer's Lease Table

**What goes wrong:** A consumer acks an ID that does not exist in its lease table (stale ack, double-ack, or ack from a different consumer's message). The broker deletes a non-existent key (no-op in Go map) and still replenishes credit, expanding the consumer's window beyond C.

**Why it happens:** No validation of `ack_id` against the lease table before processing.

**How to avoid:** Before `delete(leases, id)` and `sem <- struct{}{}`, check `if _, ok := leases[id]; !ok { /* log and skip credit replenish */ }`. This is a security/correctness guard. The test `TestMQ_NoOverPull` catches the symptom (> C unacked).

**Warning signs:** `TestMQ_NoOverPull` fails because the consumer has more in-flight messages than the initial credit.

### Pitfall 4: `make proto` Fails Because protoc Plugin Not on PATH

**What goes wrong:** `make proto` invokes `protoc --go_out ...` which internally executes `protoc-gen-go`. If `$GOPATH/bin` is not on `PATH`, protoc cannot find the plugin binary and fails with "protoc-gen-go: program not found or is not executable".

**Why it happens:** The Makefile does not prepend `$GOPATH/bin` to PATH. [CITED: /Users/ajitg/workspace/vantage/Makefile:37-40]

**Current status:** Both `protoc-gen-go` and `protoc-gen-go-grpc` are installed at `/Users/ajitg/go/bin/`. If `PATH` includes `~/go/bin`, `make proto` works. If not, prefix: `PATH="$(go env GOPATH)/bin:$PATH" make proto`.

**Warning signs:** `protoc: --go_out: protoc-gen-go: program not found or is not executable`. Resolve with `make tools` (reinstalls) or by setting PATH.

### Pitfall 5: ConsumeRequest vs ConsumeClientMsg Confusion in Smoke Tests

**What goes wrong:** `mqprobe` and any gRPC client that used `client.Consume(ctx, &pb.ConsumeRequest{})` will fail to compile after the proto change, because `Consume` now takes a bidi stream (no request message). The generated client method changes from `Consume(ctx, *ConsumeRequest)` to `Consume(ctx)` returning a bidi stream client.

**Why it happens:** Server-stream and bidi-stream have different Go client signatures in grpc-go generated code.

**How to avoid:** After regenerating stubs (`make proto`), update `mqprobe` and all tests that call `client.Consume(...)`. The new pattern:
```go
// Old (server-stream):
stream, err := client.Consume(ctx, &pb.ConsumeRequest{ConsumerId: "smoke"})

// New (bidi-stream):
stream, err := client.Consume(ctx)  // returns pb.MQService_ConsumeClient
// Then send initial credit:
stream.Send(&pb.ConsumeClientMsg{Credit: 20, ConsumerId: "smoke"})
// Then recv loop concurrently while sending acks
```

**Warning signs:** Compile error `too many arguments in call to client.Consume` or `cannot use *pb.ConsumeRequest as type...`.

### Pitfall 6: Smoke Script `sed` Parsers Break on New Counter Names

**What goes wrong:** `phase01-mq.sh` parses `consumed_total` from the inspect JSON. After D-09, `consumed_total` semantics change (now acks, not Sends). Assertions that were valid before (`consumed_total >= N after N Sends`) may fail because acks only arrive after mqprobe acks, which the current smoke implementation does not do.

**Why it happens:** The smoke script was written for the old at-most-once contract. It must be updated to send acks via bidi stream and parse the new counter names.

**How to avoid:** Update `phase01-mq.sh` to: (1) assert `delivered_total >= N` (messages sent by broker), (2) assert `consumed_total >= N` (acks after mqprobe acks each received message), (3) assert `redelivered_total = 0` (no disconnect during happy-path smoke).

### Pitfall 7: Requeue Under Buffer-Full Conditions

**What goes wrong:** If the ring is full when Requeue is called, the naive front-insertion drops the existing tail. If multiple consumers disconnect simultaneously with large lease tables, many messages may be dropped from the tail to make room. This is a legitimate trade-off (the ring is bounded by design), but the drop counter will spike.

**Why it happens:** Bounded ring + Requeue + buffer-full = tail eviction. The `dropped` counter will include these, potentially alarming operators.

**How to avoid:** Size `MQ_BUFFER_SIZE` generously relative to `credit × K_consumers`. For testing, always use `bufferSize >> credit` (like existing tests use `bufferSize = N * 2`). Document in config that `MQ_BUFFER_SIZE` should exceed `MQ_CONSUME_CREDIT × (expected concurrent consumers)`.

## Code Examples

### Smoke Probe — New Bidi Consume Pattern

```go
// Source: [ASSUMED] - derived from current mqprobe and new bidi contract

// consume opens a bidi stream, sends initial credit, receives N messages
// (acking each), then closes the stream.
func consume(ctx context.Context, client pb.MQServiceClient, n int, credit int32) (int, error) {
    stream, err := client.Consume(ctx) // bidi: no initial request message
    if err != nil {
        return 0, fmt.Errorf("open consume stream: %w", err)
    }

    // Send initial credit
    if err := stream.Send(&pb.ConsumeClientMsg{
        Credit:     credit,
        ConsumerId: "smoke",
    }); err != nil {
        return 0, fmt.Errorf("send initial credit: %w", err)
    }

    got := 0
    for got < n {
        msg, err := stream.Recv()
        if err != nil {
            return got, fmt.Errorf("recv after %d/%d: %w", got, n, err)
        }
        if msg.GetMetricName() == "" {
            return got, fmt.Errorf("received message with empty metric_name after %d/%d", got, n)
        }
        // Ack: replenishes one credit in the broker
        if err := stream.Send(&pb.ConsumeClientMsg{AckId: msg.GetId()}); err != nil {
            return got, fmt.Errorf("send ack for id %d: %w", msg.GetId(), err)
        }
        got++
    }
    return got, stream.CloseSend()
}
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Server-stream `Consume`, at-most-once | Bidi `Consume` with per-message ack, at-least-once | Phase 01.1 (this phase) | Eliminates silent loss on consumer disconnect |

- `workCh chan *pb.TelemetryMessage`: eliminated by this phase — at-most-once buffer is the bug, not a feature.

   - What we know: D-09 mentions "optionally an `in_flight` gauge". The lease table is per-consumer.
   - What's unclear: Computing a global `in_flight` sum requires either atomic aggregation per Consume handler or a shared counter.
   - Recommendation: Add `atomic.AddInt64(&s.inFlight, 1)` on each Send, `atomic.AddInt64(&s.inFlight, -1)` on each ack and on each re-enqueue. This is a global gauge. Expose it in InspectResponse as `in_flight`.

3. **How does the late-join smoke scenario work after this phase?**
   - What we know: `phase01-mq.sh` tests late-join (produce then consume as separate processes). Under bidi, the consumer must send initial credit before messages flow.
