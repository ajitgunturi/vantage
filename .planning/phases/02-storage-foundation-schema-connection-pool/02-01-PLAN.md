<objective>
Deliver the PostgreSQL storage foundation as a single thin vertical slice: a versioned migration
creates the long/narrow `gpu_metrics` time-series table (composite index + natural-key unique
constraint), and `pkg/db` exposes a shared `pgxpool` initializer plus a programmatic migration
runner that both the Collector (Phase 3) and the API Gateway (Phase 4) will import. Index usage is
proven at representative scale via a real testcontainers Postgres seeded to 100k rows.

This is a storage FOUNDATION phase — the "vertical slice" is schema → migration applied → pgxpool
opens → a real round-trip insert+query proven, not UI→API→DB. Nothing reads or writes the table at
runtime this phase; we build the seam everyone else plugs into.

Purpose: Satisfy DB-01..DB-04 and unblock Phase 3 (Collector writes) and Phase 4 (Gateway reads).
Output: `pkg/db` (config, pool, migrate), the embedded SQL migration, and the testcontainers
integration suite proving all four requirements.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| env → process | `VANTAGE_DB_DSN` (DB credentials) crosses from the environment into the process at startup |
| process → PostgreSQL | pgxpool issues SQL over the wire to Postgres; query inputs cross this boundary |
| migration source → DB | embedded SQL files are applied to the live schema via golang-migrate |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-02-01 | Information Disclosure | DSN handling in `FromEnv`/`New` error paths | high | mitigate | DSN read only from `VANTAGE_DB_DSN`; never hardcoded; error strings carry context ("db: VANTAGE_DB_DSN is required") but NOT the DSN value or password (ASVS V8) |
| T-02-02 | Tampering | SQL execution via pgxpool | high | mitigate | pgx parameterized queries ($1,$2) only; seedRows + constraint tests use bound params; no string-concatenated values into SQL (ASVS V5) |
| T-02-03 | Denial of Service | `MaxConns` pool sizing | medium | mitigate | Default pgxpool sizing (max(4,NumCPU)); capped/overridden only via `VANTAGE_DB_MAX_CONNS`; no unbounded pool |
| T-02-04 | Denial of Service | concurrent migration on multi-service startup | low | accept | golang-migrate holds a Postgres advisory lock — only one runner proceeds, others wait; safe by library design |
| T-02-05 | Tampering | destructive migration on startup | high | mitigate | Only forward `m.Up()` is run programmatically; `down.sql` is never auto-applied; `migrate.ErrNoChange` treated as success so restarts are non-destructive and idempotent |
| T-02-SC | Tampering | go get of pgx/migrate/testcontainers | low | accept | All four packages verified on proxy.golang.org with valid VCS origins (RESEARCH.md Package Legitimacy Audit — all OK/Approved); Go ecosystem, not npm/pip/cargo, so no [ASSUMED]/[SUS] human checkpoint required |
</threat_model>
